## 🧠 THE GOLDEN RULES
```
✅ Always make a new branch for every feature
✅ Always pull before you start working
✅ Always write clear commit messages
✅ Never push directly to main
✅ Never merge your own Pull Request
❌ Never work on each other's branch
```

---

## 🌿 UNDERSTANDING BRANCHES

Think of it like this:

```
main              = the official working game (sacred, don't touch directly)
feature/your-work = your personal sandbox, break whatever you want here
feature/their-work = partner's sandbox
```

Main is always clean and working.
Your branch is where the messy work happens.
When a feature is done and tested → it gets merged into main.

---

## 🚀 STARTING A NEW FEATURE (do this every time)

```bash
# Step 1 — go to main
git checkout main

# Step 2 — get latest changes (ALWAYS do this first)
git pull

# Step 3 — create your new branch
git checkout -b feature/what-youre-building

# Step 4 — open Godot and do your work...

# Step 5 — stage your changes
git add .

# Step 6 — save a checkpoint (commit)
git commit -m "describe what you just did"

# Step 7 — push to GitHub
git push -u origin feature/what-youre-building

# Step 8 — go to GitHub and open a Pull Request
```

## 💬 COMMIT MESSAGE RULES

Formula: `verb + what you did + where`

```
✅ "add hunger decay to pet stats"
✅ "fix energy bar not updating on sleep"
✅ "connect feed button to hunger stat"
✅ "add save system using JSON"

❌ "fixed stuff"
❌ "changes"
❌ "aaaaaa"
❌ "idk"
```

Why it matters — when something breaks later you can read
the commit history like a diary and find exactly when it broke.

## 🔀 WHAT IS A PULL REQUEST (PR)?

A Pull Request = you telling your partner
"hey I finished this feature, check it before we add it to main"

```
1. You finish your feature on your branch
2. Push it to GitHub
3. Go to GitHub → click "Compare & pull request"
4. Partner looks at your code changes
5. Partner clicks Merge if it looks good
   OR leaves a comment if something is wrong
6. Your code is now in main ✅
```

**Rule: Never merge your own PR. Always have partner review it first.**

## 🔄 PULLING YOUR PARTNER'S NEW CODE

When your partner merges something into main,
here's how to get it onto your branch:

```bash
# Step 1 — save your current work first
git add .
git commit -m "wip: what you were working on"

# Step 2 — go to main and pull
git checkout main
git pull

# Step 3 — go back to your branch
git checkout feature/your-branch

# Step 4 — bring main's new stuff into your branch
git merge main

# Done! Your branch now has:
# ✅ Partner's new merged stuff
# ✅ Your own work still there, untouched
```

**Pull every morning before you start working!**

## ⚠️ MERGE CONFLICTS

A conflict happens when you BOTH edited the same line in the same file.
Git doesn't know which version to keep so it asks you to decide.

Git will mark the file like this:
```
<<<<<<< your branch
max_hunger = 100
=======
max_hunger = 150
>>>>>>> partner's branch
```

How to fix it:
```
1. Open the file
2. Pick which version is correct
3. Delete the <<<, ===, >>> markers
4. Save the file
5. git add .
6. git commit -m "resolve merge conflict in pet_data"
7. git push
```

**How to avoid conflicts:** Stick to your own files.
If you need to edit a shared file, tell your partner first.

```
poundps owns: pet_stats.gd, save_load.gd, game_loop.gd
eiz owns:     hud.gd, actions.gd, sprites/
Both careful: pet_data.tres (agree on this file's structure early!)
```

## ⏪ REVERTING A BAD MERGE

If something broken got merged into main:

```bash
# Step 1 — find the bad commit ID
git log --oneline

# You'll see something like:
# a1b2c3d  merge feature/hud into main  ← bad one
# e4f5g6h  add hunger decay logic

# Step 2 — revert it (safe, doesn't delete history)
git revert a1b2c3d

# Step 3 — push
git push
```

Revert = ctrl+z that gets saved into history.
The bad commit is still there but its changes are undone.

```
Morning:

  git pull

  git merge main
  → open Godot and work

  git add .
  git commit -m "what you just did"

When feature is done:
  git push
  → open Pull Request on GitHub
  → tell partner in #progress-log on Discord
  → wait for partner to review and merge

After partner merges your PR:
  → start next feature
  → repeat from top
```

## ❓ QUICK REFERENCE

| What you want to do | Command |
|---|---|
| See what branch you're on | `git branch` |
| Create a new branch | `git checkout -b feature/name` |
| Switch to a branch | `git checkout branch-name` |
| Save your work | `git add .` then `git commit -m "message"` |
| Push to GitHub | `git push` |
| Pull latest from GitHub | `git pull` |
| Bring main into your branch | `git merge main` |
| See commit history | `git log --oneline` |
| Undo a merged commit | `git revert commit-id` |
